### Changed
- **Async full-course generation — `POST /api/v1/builder/generate-course` is now a background job.** The orchestrator makes `1 + 2 × objectives_count` sequential ~29s AI calls, so a large brief ran for minutes with the UI spinning and could lose the response even though the course committed (observed live). The endpoint now creates a job, schedules the work, and returns **`202 { job_id }`** immediately; clients poll for progress and the result.
  - **`GenerationJob`** (`app/models/generation_job.py`, `TenantScoped`) — `{status (pending|running|complete|failed), total_steps, completed_steps, phase, course_id (set on success), error (set on failure), created_by, created_at, updated_at}`. Migration `d3f5b7a9c1e4` revising head `b7d9f1a3c5e2` (offline `--sql` verified); `generation_jobs` joins the fail-closed RLS regime (`_ORG_SCOPED`, count **40 → 41**).
  - **`generate_course` progress hook** — an optional `on_progress(completed, total, phase)` async callback fired after objectives (step 1 of `1 + 2 × count`) and after each objective's lesson + assessment attempt (best-effort skips still advance). All calls occur before any DB assembly, so a progress-persisting callback only ever flushes the job row. `None` (existing callers) → identical prior behaviour.
  - **Background runner** (`app/builder/generation_runner.py::run_generation`) — runs OUTSIDE the request, so it opens its **own** org-scoped session via `org_scoped_session(org_id)` (sets the `current_org` ContextVar **and** the `app.current_org` GUC per transaction begin, mirroring `tenant_context`) — cross-tenant isolation holds. It marks the job `running`, drives `generate_course` with a progress callback that commits `completed_steps`/`phase` (so polls see live progress), and finishes `complete (course_id)` or `failed (error)`. On failure it **rolls back before marking failed**, so no orphaned partial course is ever left behind. Testability seams: `get_author_ai` (fake in CI → **zero real Anthropic calls**) and `get_generation_session_scope` (test session in CI).
  - **Status endpoint** — `GET /api/v1/builder/generate-course/jobs/{job_id}`, tenant + owner scoped (another tenant or another user → `404`). `objectives_count > 20` → `422` at body validation, before any job is created.
  - **Frontend** — `GenerateCourseStep` schedules generation, polls every ~2.5s, shows a live progress bar (`Generating… {completed}/{total} — {phase}`), drops the author into the course on `complete` (`onCreated(course_id)`), and surfaces the error with a **Retry** on `failed`. `api/builder.ts`: `generateCourse` → `{ job_id }` + new `getGenerationJob`.
  - **Tests** — backend: model tests; a `generate_course` progress-sequence test (incl. skipped-item counting); async endpoint tests (POST → poll → complete with a populated `DraftCourse` + fake-AI proof; failure → `failed` + error + **no partial course**; 422/403/503; cross-tenant + other-user `404`); a production-seam runner test exercising the real `org_scoped_session` and asserting the tenant write-stamp. Frontend: vitest poll→navigate + failure→retry. Full backend suite (918) + `npm run build` + 91 vitest green.
  - Docs: `docs/AUTHORING_AI.md` ("Full-course orchestrator").

### Added
- **Multilingual generation + session-language tutor (T3) — a course can be generated in a target language, and the tutor answers/coaches/grades in the language the learner picks at session start.** A single `language` dimension threaded through the already-governed generator + tutor seams — **no new subsystem, NO migration, NO persisted language preference.** Per `docs/specs/2026-07-07-T3-multilingual-mini-design.md` (D-a decided: learner picks at session start; session/client-held; tutor replies only — UI-chrome + course-content localization are **deferred**). **NO real Anthropic/embedding calls in CI — fakes assert the language reaches the prompt.**
  - **T3a — generation language.** `CourseBrief.language: str = "en"` (plain name / BCP-47) threads through `generate_course` + `POST /generate-course` into every per-aspect generator prompt (`generate_objectives` / `generate_item_content` / `generate_assessment` on `AIClient`/`CourseAuthorAI`) as a trailing *"Generate all content in {language}."*. The English default (`en` / `english` / blank) appends **no** instruction, so a default brief's prompts are byte-identical to pre-T3a. Author-chosen via the brief; separate from the tutor's session language.
  - **T3b (backend) — tutor language.** The tutor endpoints accept an optional `language` (`ask` / `coach` bodies; `assess` query param, default `en`) threaded through the `answer` / `coach` / `assess` orchestrators into the `answer_question` / `generate_coaching` / `evaluate_submission` seams + their pure prompt builders (`build_*_user_prompt`) as *"Respond in {language}."*. **D5 unchanged** — the language string carries no identity, so the no-identity + PII-redaction invariants hold; the grounding gate still refuses on empty retrieval **without** calling the model (language does not relax it). Only the assess **feedback** is localized — the numeric score stays language-agnostic.
  - **T3b (frontend) — session-language selector.** A `SessionLanguageProvider` (React context seeded from / written through `sessionStorage` — session/client-held, **no** backend call, **no** persisted preference) mounted at the app root; a `LanguageSelector` in the Course Player header (visible on entering a course). `TutorChat` reads the session language and sends it on every `askTutor` request (`api/tutor.ts` gains `language`, default `en`). Default English preserves current behaviour.
  - **Tests** — backend: `tests/authoring_ai/test_author_ai_client.py` (language reaches every generator prompt; default `en`/synonyms unchanged), `tests/builder/test_course_generator.py` (the brief's language reaches all `1 + 2N` generator calls; default threads `en`), `tests/tutor/test_tutor_client_language.py` (*"Respond in {language}."* in each tutor prompt; default unchanged), `tests/tutor/test_answer.py` (language reaches the seam + composed prompt; **grounding gate not relaxed**; D5 holds), API tests for `ask`/`coach`/`assess` passing the language through. Frontend: `SessionLanguage.test.tsx` (selecting a language → `askTutor` called with it; default `en`). Full backend suite + `npm run build` + vitest green.
  - Docs: `docs/AUTHORING_AI.md` (generation language) + the T3 mini-design spec.
- **Release→ingest convergence (Phase B) — a released `CurriculumVersion` now auto-builds its retrieval index.** `ingest_version` existed but nothing called it on release, so a just-released course had an empty index and the RAG tutor could not answer about it. The convergence step is now wired: when a `CurriculumVersion` becomes active, ingestion fires automatically. NO migration (`ContentChunk` already exists); NO real embedding calls in CI (FakeEmbedder default).
  - **Background runner** (`app/core/retrieval/ingest_runner.py::run_ingest`) — scheduled on a FastAPI `BackgroundTask` **after the release commits** (per design D4 — embedding a whole course is slow; a release must not block on it). It runs OUTSIDE the request, so it opens its **own** org-scoped session via `org_scoped_session(org_id)` (sets the `current_org` ContextVar **and** the `app.current_org` GUC per transaction begin, mirroring `generation_runner`) — cross-tenant isolation holds and the written `ContentChunk` rows are stamped with the releasing tenant. Idempotent per version (`ingest_version` replaces). **Never raises**: the release has already committed, so an ingest failure is logged and swallowed (the index can be rebuilt via the admin trigger). Embedder defaults to `get_embedder()` → `FakeEmbedder` in CI.

- **Proactive coaching (B4) + assessment feedback (B5) — two new tutor modes on the governed seam, backend.** Both extend the same `Tutor` seam + tutor router as B3 (which is why they ship together), reuse the tutor conversation store + `AssessmentSubmission` (**NO migration**), and carry the same **D5** posture as `answer`: the model NEVER receives learner identity, and learner free text is PII-redacted before it leaves the backend. Per `docs/specs/2026-07-07-phaseB-rag-tutor-design.md` §B. **NO real Anthropic/embedding calls in CI — a fake `tutor_ai` + `FakeEmbedder` are injected.**
  - **B4 — `coach`** (`app/core/tutor/coach.py::coach`) — assembles an **anonymized progress signal** from the learner's `LearnerProgress` (completed/total counts + the *next* uncompleted item's section + lineage label — never an id/name/email), retrieves the next objective's own `ContentChunk`s for grounding, and produces a Forge-Coach-style message (checks understanding, names ONE concrete next step). Persists a tutor turn (with citations) to `TutorConversation`/`TutorMessage`. Unlike `answer`, an empty index does NOT hard-refuse — coaching still runs on the real progress facts, with no excerpts and no fabrication (graceful degradation). Governed seam method `AIClient.generate_coaching(progress, context_chunks)` via `_parse`; the signature has **no identity parameter** (D5 by type shape). Endpoint `POST /api/v1/learn/tutor/{enrollment_id}/coach` (learner-gated, enrollment-scoped; cross-tenant/learner → 404, wrong role → 403, no key → 503).
  - **B5 — `assess`** (`app/core/tutor/assess.py::assess`) — grades an `AssessmentSubmission` against the assessment item's **rubric** (read from the item's `ContentVersion.metadata_["rubric"]`, where the authoring path stores it) + the assessment prompt, then **writes `score` (0.0–1.0) + `feedback` back onto the submission**. D5: the FULL raw `response_text` stays on the submission (secure server-side record); only `redact_pii(response_text)` reaches the model, alongside course-material rubric + prompt — no identity. A missing rubric degrades gracefully (empty rubric; graded conservatively against the prompt). Governed seam method `AIClient.evaluate_submission(rubric, assessment_prompt, response)` via `_parse`. Endpoint `POST /api/v1/learn/tutor/{enrollment_id}/submissions/{submission_id}/assess` (learner-gated, enrollment-scoped; same 404/403/503 discipline).
  - **Tests** — `backend/tests/tutor/test_coach.py`, `test_assess.py`, `test_coach_assess_api.py` (fakes + seeded data): coach uses the progress signal (assert `"1 of 3"` + next-item label reach the prompt) + is D5-anonymized (no learner/enrollment id in the prompt) + grounds in the next objective + degrades on an empty index + wraps up when all items complete; assess writes score+feedback to the submission + is D5-redacted (raw stored, `[EMAIL]`/`[PHONE]` sent) + degrades with no rubric; both endpoints role-gated + cross-tenant/enrollment 404 + 503 without a key. - **Monitor Queue page** (`/monitor-queue`, roles `architect`/`program_manager`): live view of monitoring and rejected assessments — confidence chips with delta vs. previous evaluation, sighting counts, expandable rationale/dossier, and promoted-CCR links. Backed by `GET /api/v1/freshness/assessments?recommendation=` (org-scoped, read-only v1).
- **Digest JUDGE line**: `N gaps judged: A promoted, M monitoring (K strengthened), R rejected, S re-reviewed` — added to the ACTIONS TAKEN section when `gaps_judged > 0`; absent on old runs (backward-compatible `.get`).

### Changed
- **`sync_logs.version_id`** relaxed to nullable (Phase 4): new-model releases have no legacy `Version` row. Existing legacy-sync callers still pass `version_id`; Phase 4 callers pass `version_id=None` and set `curriculum_version_id` instead.
- **`sync_logs`** gains a nullable `curriculum_version_id` FK → `curriculum_versions` (`SET NULL` on delete). This lets the Phase 4 sync service link a sync attempt directly to the immutable `CurriculumVersion` that triggered it.
- **Pipeline runner** gains a pending-sync sweep (Phase 4, step 0) and four new stats keys (`syncs_attempted/succeeded/failed/skipped`). Sweep runs before signal gathering so one run can both retry pending syncs and process new signals.
- **Pipeline runner step 5** routes every detected gap through the Judge before creating a CCR; only gated `adopt_now` outcomes (confidence ≥ threshold) become proposals. Promoted CCRs carry `assessment` (with all 7 dimension scores) + `dossier` (full multi-run evidence) + `ai_research` in `impact`. New `freshness_pipeline_runs` stats keys: `gaps_judged`, `gaps_adopted`, `gaps_monitored`, `gaps_rejected`, `gaps_resurrected`, `gaps_strengthened`.
- **Pipeline runner step 6b** (Phase 3): after enrichment, when `FRESHNESS_GENERATION_ENABLED` is `true`, generates a `ReleaseChangeSet` for each adopted CCR. Generation failure is contained per-CCR — the run never fails for generation; the CCR degrades to Phase-2 shape and `changesets_failed` is incremented.

## [1.2.0] - 2026-06-04 — Multi-tenant isolation

## [1.0.0] - 2026-06-04

First public portfolio release. CurricMesh turns a 116-page internal curriculum-governance framework into a database-backed, authenticated, role-based application that *enforces* the change-control workflow, *computes* the dependency cascade, *diffs* versions, and adds an AI layer whose quality is *measured* against planted ground truth.

### Added

**Milestone A — Core platform (workflow, RBAC, lifecycle, gate)**
- Async FastAPI backend on PostgreSQL 16 (SQLAlchemy 2.0 async, Alembic migrations), JWT auth (PyJWT + bcrypt) with constant-time login.
- Six-role RBAC: `architect`, `program_manager`, `instructor_lead`, `instructor`, `qa_lead`, `devops`.
- Semantic versioning plus a lifecycle state machine: `draft → review → approved → active → archived → sunset`, with a `review → draft` QA bounce-back.
- Change Control Request (CCR) → six-dimension QA review (`content_accuracy`, `alignment`, `prerequisites`, `consistency`, `instructor_support`, `student_experience`, scored 1–5 with evidence) → two-approver release gate.
- Release gate enforced atomically **in the engine, not the UI**: requires a passing QA review and ≥2 distinct approvers including ≥1 instructor, with author self-approval rejected and duplicate approvals blocked by a `UniqueConstraint`. Release uses `SELECT FOR UPDATE` + an idempotency guard so it can never double-activate.
- React 19 + Material UI dashboard with JWT auth, plus a History/audit timeline.

**Milestone B — Dependency cascade, diff, and notifications**
- Cycle-safe BFS dependency-cascade engine that computes the downstream version bumps a change forces.
- Version-diff engine: `difflib` for prose, structured JSON diff for rubrics and learning objectives.
- Dependency-graph visualization (reactflow) and a version-diff viewer in the frontend.
- Slack/email notifications — fired after commit, non-blocking, surface-don't-swallow.

**Milestone C — AI layer (researcher, judge, eval, inbox)**
- **SOTA-gap researcher** (Claude `claude-opus-4-8`) reads a curated synthetic industry corpus (40 entries: 20 job postings + 20 vendor docs) plus covered topics, and **drafts CCRs** for under-covered topics as a system "AI Researcher" actor — entering the normal QA/approval flow, never bypassing the gate.
- **LLM-as-judge QA** pre-scores the six QA dimensions (1–5 with evidence) as an advisory draft carrying a sentinel verdict that can never satisfy the release gate; a human QA Lead reviews and promotes it.
- **Evaluation harness** measures the AI against planted ground truth — gap-detection precision/recall vs. three planted gaps and QA-judge agreement vs. human labels (within ±1). Headline metrics: **gap precision 0.667, recall 0.667** (perfect recall on the two critical-signal gaps; honest miss of the one moderate-signal gap) and **QA-judge agreement 0.833**.
- **AI Findings Inbox** in the frontend for reviewing and promoting AI-drafted CCRs and pre-scores.
- AI sits behind a mockable Protocol seam: deterministic and offline in CI via recorded snapshots, live with `ANTHROPIC_API_KEY`.

**Tooling, CI, and seed**
- GitHub Actions CI: backend `pytest` against real PostgreSQL 16 with an Alembic `upgrade → downgrade` round-trip check; a **non-blocking** AI-eval job that publishes a markdown report artifact; frontend `npm ci` against a Linux-generated lockfile behind an `npm audit` high/critical gate. Read-only CI tokens.
- 213 backend tests (pytest + pytest-asyncio) and 45 frontend tests (Vitest).
- Demo seed: 1 organization, 6 role users (`<role>@curricmesh.demo` / `demo-pass-123`), a 12-week "Agentic AI Architecture in Production" bootcamp, 4 projects, 41 assets, 1 active cohort. Load with `python -m seed.bootcamp_curriculum`; corpus with `python -m seed.load_sota`.

### Notes

CurricMesh is a portfolio and evaluation artifact — not a production service. It ships with **synthetic data only** and no real secrets. See [SECURITY.md](SECURITY.md) for the data-handling posture and production-deployment obligations.
